下面给你一份 **Claude Code CLI 的 slash 命令对照表 + 实战最佳实践**（偏工程师视角，结合你这种做 vLLM / 推理系统的场景会更实用）。

---

# 🧾 一、常用 Slash 命令总览

> 不同版本可能略有差异，但核心思路一致

## 🧠 1. 模型 &模式控制

### `/model`

**作用：切换模型**

```bash
/model
```

👉 交互式选择（如 opus / sonnet 等）

**最佳实践**

* 复杂设计 → 用更强模型（opus）
* 日常 coding → 用 sonnet（更快更省）

---

### `/buddy`

**作用：进入协作（pair programming）模式**

👉 让 Claude：

* 主动拆任务
* 连续推进
* 给 next step

**最佳实践**

* 重构 vLLM / 推理框架时开启
* Debug 多模块问题时开启
* ❌ 简单问答别开（会变啰嗦）

---

### `/plan`

**作用：先做规划，不直接写代码**

👉 输出：

* 分步骤方案
* 架构设计
* 风险点

**最佳实践**

```text
/plan 实现一个 KV cache 跨节点传输方案
```

✔ 强烈推荐在以下场景先用：

* 新系统设计
* 大功能开发
* 性能优化方案

---

## 💻 2. 代码生成 & 修改

### `/edit`

**作用：修改已有代码（核心命令）**

👉 会：

* 理解当前文件
* 生成 patch / diff
* 保持上下文一致

**最佳实践**

```text
/edit 把这个函数改成支持 batch 推理
```

✔ 优点：

* 比“直接粘代码”更安全
* 不容易破坏已有逻辑

---

### `/fix`

**作用：修 bug / 修报错**

```text
/fix 这个编译错误
```

👉 会：

* 分析 error log
* 定位问题
* 给修复方案

**最佳实践**

* CUDA / 编译错误
* Python import / 环境问题
* vLLM build 失败

---

### `/explain`

**作用：解释代码**

```text
/explain 这段 CUDA kernel 在干嘛
```

👉 输出：

* 逐步解释
* 数据流
* 性能关键点

---

## 🔍 3. 分析 &理解

### `/review`

**作用：代码 review**

👉 会检查：

* bug
* 性能问题
* 可读性
* 潜在风险

**最佳实践**

```text
/review 这个 attention kernel
```

---

### `/analyze`

**作用：深度分析（比 explain 更偏“工程”）**

👉 用于：

* 性能瓶颈
* 架构问题
* 推理链路分析

---

## ⚙️ 4. 项目级操作

### `/files`

**作用：查看当前上下文文件**

👉 类似：

* 当前 workspace 有哪些文件
* Claude 能看到什么

---

### `/add`

**作用：把文件加入上下文**

```text
/add vllm/kv_cache.py
```

👉 很关键！否则模型“看不到代码”

---

### `/drop`

**作用：移除上下文文件**

---

## 🧪 5. 测试 &执行

### `/test`

**作用：生成测试代码**

```text
/test 给这个模块写单测
```

---

### `/run`

**作用：运行命令 / 代码（部分版本支持）**

---

## 🧰 6. 其他实用命令

### `/clear`

清空上下文（类似 reset）

---

### `/help`

查看所有命令

# 🚀 二、推荐工作流（非常重要）

结合你做推理系统，这套 workflow 很实用：

---

## 🧩 场景1：做一个新功能（比如 KV 传输）

```text
/plan 设计 KV cache 跨节点传输（RDMA / Mooncake）
```

⬇

```text
/buddy
```

⬇

```text
/edit 实现第一版接口
```

⬇

```text
/review
```

⬇

```text
/analyze 性能瓶颈
```

---

## 🐛 场景2：排查 bug（你之前那种 vLLM 编译问题）

```text
/fix 报错日志
```

⬇

```text
/explain 根因
```

⬇

```text
/edit 修复代码
```

---

## ⚡ 场景3：优化性能（GPU / KV cache）

```text
/analyze 当前实现瓶颈
```

⬇

```text
/plan 优化方案
```

⬇

```text
/edit 实现优化
```

⬇

```text
/review 检查副作用
```

---

# 🧠 三、高阶技巧（很少有人会用）

## 💡 1. `/plan` + `/edit` 组合

👉 避免“直接生成垃圾代码”

错误方式：

```
直接让它写
```

正确方式：

```
/plan → 确认 → /edit
```

---

## 💡 2. `/add` 是性能关键

👉 Claude 看不到代码 = 瞎猜

✔ 正确：

```text
/add vllm/worker.py
/add kv_connector.py
```

---

## 💡 3. `/buddy` 控制复杂度

* 开：复杂任务
* 关：精确改代码

---

## 💡 4. 小步提交

不要：

```
改一整个系统
```

要：

```
每次 /edit 一个模块
```

---

# ⚠️ 四、常见坑

❌ 1. 不用 `/add`
→ 模型胡编

❌ 2. 一上来就 `/edit`
→ 设计混乱

❌ 3. `/buddy` 一直开
→ 输出过多

❌ 4. 不 `/review`
→ 引入隐性 bug

---

# 🧭 一句话总结

👉 Claude Code CLI 的核心不是“让它写代码”，而是：

> **用 `/plan` 做架构，用 `/edit` 精准改代码，用 `/buddy` 推进复杂任务**
