已经在vllm社区待过很久了，一直见证社区的发展。首先和谐共创肯定是社区的宗旨，我没见过比vllm更包容的社区。

1. 我觉得这个确实需要社区管理者（比如游神）想想怎么解决。毕竟最终他们是需要对代码负责的人。
2. 其实社区代码合入，在历史上有很多种解决办法。比如trtllm，他们之前从来不merge代码，都是看每个pr，然后选择性拿几个，之后改改，最后新版本一发，把所有开着的pr都关了。欢迎下次投递。也可以选择通过类似于apache的社区投票，只有三票以上通过才能合，管理层带一票否决权。
3. 我觉得代码提交的门槛要提高。比如小问题（文档，typo，注释）只允许正式贡献者修。任何人想要成为正式贡献者，得去社区领任务，去rfc选最终被社区采纳。rfc会准备一些大一点的活，检查大家能否做好。非rfc问题，除非是大bug（比如开issue被社区确认的），才能采纳
4. 关于pr是否是由人写的并不重要。但是提交人起码能够完全解释清楚为什么这么做。每一条反馈都得手写原因，但凡有ai味复制粘贴直接警告三连，不改的关issue。
5. 大家算力这么多也可以捐，vllm开启募捐渠道，大家把算力捐出来用于看pr，修改bug，改ci之类的，可以创建另一套奖励机制。
6. 最终其实真正工程完美的开源项目都是独裁。（欢迎各位同学举个反例）从linus大喷子，到后面别的项目。得有一些人把控主线，迭代快。**社区更希望的是有长期贡献者，不仅能够开发新功能，同时能够继续维护这一块3年五年。现在太多想干一票镀金就跑的人了，不合入哪怕很好的也是情理之中。**
