set_default_torch_num_threads 是一个临时生效的 context manager，逻辑如下：

```python
# 进入时：将 torch 线程数设为 num_threads（默认读 OMP_NUM_THREADS，否则 1）
torch.set_num_threads(num_threads)
yield  # 执行多模态处理
# 退出时：恢复原来的线程数
torch.set_num_threads(old_num_threads)
```

为什么 online serving 要限制到 1：多请求并发时，每个请求的多模态处理（图片 resize、特征提取等 CPU 密集型操作）都想抢占更多 CPU 线程，导致 OpenMP 线程池过度订阅（over-subscription），反而降低整体吞吐。

---

**Online Serving 为什么要限制？**

Online serving（API server）中，多个请求同时触发多模态处理：

```
Request 1 → _process_multimodal → torch 抢占 N 个 CPU 线程
Request 2 → _process_multimodal → torch 抢占 N 个 CPU 线程
Request 3 → _process_multimodal → torch 抢占 N 个 CPU 线程
...
```

如果每个 _process_multimodal 调用都用 N 线程，总线程数 = N × 并发请求数，远超 CPU 核心数，导致：

OpenMP 线程池过度订阅：大量线程上下文切换
各请求互相争夺 CPU，整体吞吐反而下降
限制为 1 线程/请求 → 总线程数 = 并发数，可控

---

**Offline 模式为什么不需要限制？**

Offline 的 LLM 类是顺序处理（sequential），同一时刻只有一个请求在做多模态处理：

```
Request 1 → _process_multimodal → 处理完毕 →
Request 2 → _process_multimodal → 处理完毕 →
Request 3 → _process_multimodal → ...
```

没有并发竞争，所以：

使用更多线程 = 在单个任务内并行化 CPU 工作（如图片 decode、resize、特征提取）
没有其他请求争抢，不会 over-subscription
更多线程直接带来单请求更快的处理速度
