## Perf Tuning

我在 vllm 上基于 https://github.com/vllm-project/vllm/issues/38175 为 DeepSeek-OCR 实现了 ViT Cuda graph 的支持。然而，我发现开启 ViT Cuda graph 之后，性能比 eager（不开）时还差，请分析当前实现存在的问题，有哪些地方可以改进从而优化性能。
参考资料：https://github.com/vllm-project/vllm/pull/42796

---
**将local patch从cuda graph中拿出来后，对于global image，能不能将newline token的拼接从postprocess_encoder_output()中放到encoder_cudagraph_forward()中？能或不能的原因是什么？**

Global image 的 newline 拼接完全满足 CUDA graph 的三个关键约束：

1. Shape 是编译期常量 — 每张 global image 固定产出 16×16=256 tokens，加 newline 列后 16×17=272 tokens。image_side=16 是模型结构决定的硬编码值（BASE_SIZE//patch_size//downsample_ratio = 1024//16//4 = 16），不随输入图片变化。
2. 无数据依赖分支 — 操作序列是纯代数变换：reshape → expand → cat → flatten，没有 if 分支。self.image_newline 是 nn.Parameter（leaf tensor），在 graph 捕获时可正常访问。
3. batch 维度固定 — graph 捕获时 pixel_values shape 为 [max_batch_size, 3, 1024, 1024]，replay 时通过 zero-padding 保持 shape 不变，newline expand 的目标 shape [max_batch_size, 16, 1, n_embed] 也是固定的。

对比 local patch：local patch 的 tile grid 组装需要 crop_shape（每张图不同），且 patches 数量可变，因此无法放入 CUDA graph。Global image 不存在这些问题。

---
**在postprocess_encoder_output()中，既然local patch的计算已经完全是eager计算了，能不能将其中local patch计算和newline token添加的逻辑直接替换为调用_encode_local_features()方法？**

---
**当输入图像大小为(896, 896)时（即会做tile，生成global patch），开启vit cuda graph，deepseek-ocr推理的性能反而下降了，是什么原因导致的？还能如何优化？**
